⚙️ Request Lifecycle

Requests enter via standard HTTP but are immediately processed by a custom Middleware Stack. Notably, ThreadLocalMiddleware acts as a bridge, capturing the authenticated user from the request and making it available globally for signals and decorators deep within the application stack where the request object isn't natively accessible.

🔒 DRF Security Layers

Our security is multi-layered. SimpleJWT provides stateless authentication via Bearer tokens. StrictDjangoModelPermissions ensures that even standard Django permissions are applied to GET requests (which Django skips by default). Finally, Scoped Throttling provides granular protection for sensitive actions like inmate transfers and medical record access.

⚡ The Audit Pipeline & Safety Engine

We use a hybrid approach to auditing. Django Signals (post_save, post_delete) automatically track any database mutations. For read operations, we use a custom @audit_read decorator. The Transfer Safety Engine inside InmateViewSet actively parses these generated logs to permit or deny cell block transfers based on historical physician activity.

🛡️ The Admission Engine

A two-step ingestion process. First, the View checks facility-wide rules (Cell Block Capacity evaluation). Secondly, the Serializer handles low-level data integrity constraints using transaction.atomic() to write both the `InmateProfile` and `MedicalFile` simultaneously, guaranteeing no ghost profiles are created.
